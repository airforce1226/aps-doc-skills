# 문서 유형별 구성 및 작성 원칙

이 파일은 프로젝트 문서를 작성할 때 참고하는 **표준 목차**와 **작성 원칙**을 담는다.
SKILL.md 의 절차에 따라, 아래 목차를 토대로 사용자와 협의하거나 기존 초안을 채워 spec JSON 을 만든다.

---

## 공통 작성 원칙

1. **빈칸은 만들지 말고 "TBD"로 둔다.** 모르는 값은 추측하지 말고 spec 의 셀/문구에 `"TBD"`(또는 `(미정 …)`)로 둔다. 엔진이 회색 이탤릭으로 표시한다.
2. **상대 날짜는 절대 날짜로.** "다음 달" → "2026년 7월"처럼 변환한다.
3. **하나의 개념은 하나의 단어로.** (예: user ↔ member 혼용 금지)
4. **부정형 회피.** `유효함`을 쓰고 `유효하지않음`은 호출부에서 처리.

## 독자에 맞춘 어조 (중요)

문서를 읽을 대상을 먼저 확인한다.

- **임원/경영진 대상**: 전문 용어를 최대한 줄인다. 기술 스택·아키텍처 용어(SPA, REST API,
  WebForms, 스프린트/백로그/Increment, ERD, UAT 등)는 **효과 중심의 일상 표현**으로 바꾸거나
  쉬운 설명을 덧붙인다. 구체 기술명은 "참고용 표 한 곳"에만 남긴다.
  - 예: "ASP.NET WebForms 모놀리식" → "오래된 개발 기술로 구축된 노후 시스템"
  - 예: "단위/통합 테스트, UAT" → "기능별 점검 / 전체 통합 점검 / 현업 검증"
  - 예: "데이터 이관(MySQL→PostgreSQL)" → "기존 자료 이전"
- **개발자/실무 대상**: 기술 용어를 그대로 사용해도 된다. 개발자용 메모(연동 방식, 확인 필요
  사항 등)는 임원 보고서에서는 제외한다.

---

## 유형 1: 착수 보고서 (kickoff report)

시스템 전환·구축 프로젝트의 시작을 알리는 보고서. 보통 임원 대상이다.

표준 목차:
1. **사업 개요** — 사업명 / 사업 기간 / 착수일 / 사업 목적 / 대상 사용자
2. **전환 필요성 및 기대효과** — (왜 하는가)
   - 2.1 현행 시스템 현황 및 문제점 (As-Is)
   - 2.2 전환 후 기대효과 (To-Be / 무엇이 좋아지는가)
3. **추진 범위 및 대상 시스템** — 업무 범위 / 운영 환경 / 연계 시스템 / 대상 메뉴(담당자 표)
4. **목표 시스템 구성 및 적용 기술** — 구성도 / 적용 기술(현행→전환 표)
5. **추진 일정 및 개발 방식** — 개발 방식 / 진행 절차 / 단계별 산출물
6. **수행 조직 및 인력 투입 계획** — 조직·역할 / 투입 현황(참여율·기간)
7. **품질 관리 및 리스크 관리 계획** — 점검 계획 / 리스크 대응 표

핵심 입력값: 사업 기간·착수일, 현행/목표 기술, 대상 메뉴와 담당자, 개발 방식(워터폴/애자일).

## 유형 2: 요구사항 정의서 (requirements)

1. 문서 개요 (목적/범위/용어)
2. 현행 업무 분석
3. 기능 요구사항 (기능 목록 표: ID / 기능명 / 설명 / 우선순위)
4. 비기능 요구사항 (성능·보안·가용성 등)
5. 인터페이스/연계 요구사항
6. 제약 사항 및 가정

## 유형 3: 회의록 (meeting notes)

1. 회의 정보 (일시 / 장소 / 참석자 / 작성자)
2. 안건
3. 논의 내용
4. 결정 사항
5. 액션 아이템 (담당자 / 기한 표)

---

## spec JSON 작성 요령

- 표는 `{"table": {"headers": [...], "rows": [...], "widths": [cm...]}}` 형태.
- 현행→전환 비교처럼 열이 많으면 widths 로 폭을 조절한다(A4 본문 합계 약 15.5cm).
- 참고/주석 문단은 `{"para": "※ ...", "grey": true, "size": 9}` 로 작게 표시.
- 라벨이 있는 개요 항목은 `{"bullet": "값", "label": "사업명"}`.
